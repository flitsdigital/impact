## Why this matters

This repo has **zero tests, no test runner, no CI, and no `typecheck`
script** — the only verification is `next build`, run manually. Git history
shows the cost: date/timezone bugs in the Gantt view were fixed twice
(commits `4c76388` "DST-safe addDays" and `89f5f58` "local date for
today-line ... avoid UTC timezone shift"). Those helpers are pure functions —
the cheapest possible thing to test. This plan adds the missing scripts,
vitest, characterization tests for the date helpers (locking in the DST fixes),
and a GitHub Actions workflow so every push gets typecheck + test + build.
Several other plans (004, and any future refactor of the 723-line
ProjectDetailModule) assume this baseline exists.

```ts
function endOfMonth(d: Date): Date {
  return new Date(d.getFullYear(), d.getMonth() + 1, 0, 23, 59, 59, 999)
}

function addDays(dateStr: string, days: number): string {
  const [y, m, d] = dateStr.split('-').map(Number)
  const date = new Date(y, m - 1, d)   // local midnight — DST-safe
  date.setDate(date.getDate() + days)
  return `${date.getFullYear()}-${String(date.getMonth() + 1).padStart(2, '0')}-${String(date.getDate()).padStart(2, '0')}`
}

function toPct(dateStr: string, rangeStart: Date, totalMs: number): number {
  return ((new Date(dateStr).getTime() - rangeStart.getTime()) / totalMs) * 100
}

function fmtDate(dateStr: string): string {
  return new Date(dateStr).toLocaleDateString('nl-NL', { day: 'numeric', month: 'short' })
}

function toLocalDateStr(d: Date): string {
  return `${d.getFullYear()}-${String(d.getMonth() + 1).padStart(2, '0')}-${String(d.getDate()).padStart(2, '0')}`
}
```

### Step 3: Write characterization tests

Create `vitest.config.ts`:

```ts
import { defineConfig } from 'vitest/config'
import path from 'node:path'

export default defineConfig({
  test: { environment: 'node', include: ['**/*.test.ts'], exclude: ['node_modules', '.next'] },
  resolve: { alias: { '@': path.resolve(__dirname) } },
})
```

Create `lib/dates.test.ts` covering (the runner's local timezone is what the
app runs in; the team is in Europe/Amsterdam — set `TZ` in CI, see Step 4):

- `addDays` happy path: `addDays('2026-06-10', 5)` → `'2026-06-15'`
- month boundary: `addDays('2026-01-31', 1)` → `'2026-02-01'`
- year boundary: `addDays('2025-12-31', 1)` → `'2026-01-01'`
- leap day: `addDays('2024-02-28', 1)` → `'2024-02-29'`
- negative offset: `addDays('2026-03-01', -1)` → `'2026-02-28'`
- **DST regression (the bug fixed in `4c76388`)** — spring-forward in
  Europe/Amsterdam (2026-03-29) and fall-back (2026-10-25):
  `addDays('2026-03-28', 1)` → `'2026-03-29'`, `addDays('2026-03-28', 2)` →
  `'2026-03-30'`, `addDays('2026-10-24', 2)` → `'2026-10-26'`
- `toLocalDateStr(new Date(2026, 0, 5))` → `'2026-01-05'` (zero-padding)
- `endOfMonth(new Date(2026, 1, 10))` → Date with `.getDate() === 28`,
  `.getMonth() === 1`
- `toPct('2026-01-16', new Date('2026-01-01'), 30 * 86_400_000)` → `50`
  (UTC-parsed midpoint of a 30-day range — characterize current behavior,
  do not "fix" the UTC parse)

Use plain `describe`/`it`/`expect` from `'vitest'`.

**Verify**: `TZ=Europe/Amsterdam pnpm test` → all tests pass (≥12 assertions);
then `TZ=UTC pnpm test` → also passes (the helpers are TZ-safe by design — if
a case fails under UTC, that's a real finding: STOP and report which one).

### Step 4: CI workflow

Create `.github/workflows/ci.yml`:

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  verify:
    runs-on: ubuntu-latest
    env:
      TZ: Europe/Amsterdam
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck
      - run: pnpm test
      - run: pnpm build
        env:
          NEXT_PUBLIC_SUPABASE_URL: https://placeholder.supabase.co
          NEXT_PUBLIC_SUPABASE_ANON_KEY: placeholder
      - run: pnpm lint
        continue-on-error: true   # 36 pre-existing problems — informational until lint debt is paid
```

Note: `pnpm build` may genuinely need valid-looking env vars only at runtime,
not build time — if the build fails on the placeholders with a Supabase
connection error, STOP and report (the page may be doing build-time fetching
that should be dynamic).

**Verify**: `cat .github/workflows/ci.yml` matches; locally run the same
sequence: `pnpm typecheck && pnpm test && pnpm build` → exit 0.
